# plain-language-brief

## Goal

Convert internal UX findings into language the client can understand.

## When to use

Use after analysis and before final user-facing output.

## Process

1. List internal findings.
2. Replace expert terms using the Frameward terminology map.
3. Explain each change as a user benefit.
4. Avoid acronyms unless the user used them first.
5. Keep tone practical and confident.

## Output

```md
### What I understood
-

### What I changed or will change
-

### Why this helps
-

### What remains
-
```

## User-language rule

Do not expose expert UI/UX terminology to non-expert users. Translate internal findings into plain language.
